# UrbanFlow Nexus: Technical Design Document

## 1. System Architecture Overview

UrbanFlow Nexus is a prescriptive traffic intervention platform built on a microservice-style monolithic backend.
- **Frontend**: Next.js (React, TypeScript, TailwindCSS) for the Command Center UI.
- **Backend**: FastAPI (Python 3.10+) for asynchronous API handling.
- **Data Stores**: PostgreSQL (Relational/Spatial data) and Neo4j (Intervention Knowledge Graph).
- **Compute Engines**: 
  - Tabular ML: CatBoost for categorical prediction.
  - Simulation: NetworkX for in-memory graph traversal.
  - Optimization: Google OR-Tools for integer programming (resource allocation).
  - Generative AI: LangChain + OpenAI for operational brief generation.

## 2. Database Schema (PostgreSQL)

### 2.1 Table: traffic_events
Stores the raw incident data and CatBoost predictions.
- `id` (UUID, Primary Key)
- `event_type` (String) - e.g., 'Breakdown', 'Rally', 'Accident'
- `priority` (String) - 'High', 'Medium', 'Low'
- `zone` (String) - e.g., 'Koramangala', 'Central'
- `road_closure` (Boolean)
- `predicted_severity` (Float) - Output from CatBoost
- `predicted_resolution_time_mins` (Integer) - Output from CatBoost
- `created_at` (Timestamp)

### 2.2 Table: simulated_interventions
Logs the outcome of the Counterfactual Simulator and OR-Tools decision.
- `id` (UUID, Primary Key)
- `event_id` (UUID, Foreign Key)
- `strategy_name` (String)
- `officers_deployed` (Integer)
- `barricades_deployed` (Integer)
- `impact_reduction_pct` (Float)
- `ties_score` (Float)
- `is_final_selection` (Boolean)

## 3. Knowledge Graph Schema (Neo4j)
The graph maps historical events to interventions to find past successes.
- **Nodes**:
  - `(Event {id, type, zone, priority})`
  - `(Intervention {id, strategy_name, officers, barricades})`
- **Relationships**:
  - `(Event)-[:MITIGATED_BY {impact_reduction: Float, success_flag: Boolean}]->(Intervention)`
- **Cypher Query Goal**: Find interventions connected to similar historical events where `success_flag = true`, ordered by `impact_reduction` descending.

## 4. Machine Learning & Simulation Engines

### 4.1 Tabular Predictor (CatBoost)
- **Input**: Categorical features from `traffic_events` (`event_type`, `priority`, `zone`, `road_closure`).
- **Target 1**: `predicted_severity` (Regression: 1.0 to 10.0 scale).
- **Target 2**: `predicted_resolution_time_mins` (Regression: Minutes).
- **Behavior**: Models are pre-trained on `dataset.csv` and loaded into memory via `.cbm` files during FastAPI startup.

### 4.2 Counterfactual Simulator (NetworkX)
- **Graph Initialization**: A static road network is loaded into NetworkX (Nodes = Junctions, Edges = Corridors with capacity weights).
- **Baseline ($do(X=\emptyset)$)**: Calculate network delay when a specific edge capacity drops to 0 due to the event.
- **Intervention ($do(X=\text{Intervention})$)**: Re-calculate network delay assuming intervention features.
- **Output**: Delta Impact (Percentage reduction in delay).

### 4.3 Optimization Engine (OR-Tools)
Calculates the Traffic Intervention Effectiveness Score (TIES).
- **Formula**: `TIES_k = Delta_Impact_k / Sum(w_j * R_{kj})`
  - `w_j`: Operational weight (e.g., Officer = 5.0, Barricade = 1.0)
  - `R_{kj}`: Quantity of resources used.
- **OR-Tools Objective**: Maximize `TIES_k` subject to `Total Officers <= Available Officers` and `Total Barricades <= Available Barricades`.

## 5. API Contracts (FastAPI)

### 5.1 POST /api/v1/event/ingest
Ingests a new event, predicts severity, and saves to DB.
**Request**:
```json
{
  "event_type": "Breakdown",
  "priority": "High",
  "zone": "Central",
  "road_closure": true
}
```
**Response (200 OK)**:
```json
{
  "event_id": "uuid-1234",
  "predicted_severity": 8.5,
  "predicted_resolution_time_mins": 45
}
```

### 5.2 POST /api/v1/simulate
Runs the Knowledge graph query, NetworkX simulation, OR-tools optimization, and OpenAI brief generation.
**Request**:
```json
{
  "event_id": "uuid-1234",
  "available_officers": 20,
  "available_barricades": 10
}
```
**Response (200 OK)**:
```json
{
  "recommended_strategy": "Deploy Officers + Upstream Diversion",
  "resources_allocated": { "officers": 4, "barricades": 2 },
  "ties_score": 12.5,
  "expected_impact_reduction_pct": 42.0,
  "operational_brief": "Markdown string generated by LangChain..."
}
```

## 6. Testing Strategy & Guardrails
- All business logic must be pure functions where possible to allow property-based testing (`hypothesis`).
- Network calls must be stubbed: OpenAI and Neo4j calls must use dependency injection or mocking in `tests/` so CI runs instantly and deterministically.
- Edge Cases: Handle 0 available resources gracefully. Handle unknown `event_type` categories gracefully in CatBoost (fallback to median predictions).
